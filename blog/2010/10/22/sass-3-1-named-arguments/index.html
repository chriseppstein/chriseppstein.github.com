What’s New in Sass 3.1: Named Arguments

Rationale

If you’ve ever had a mixin that needed to accept many arguments for customization, but you wanted to mostly rely on the default values for those arguments then you’ve probably found yourself frustrated. To make this situation easier to manage, Sass is introducing Named Arguments in Sass 3.1. Named arguments are of the form $name: value and must come after any normal (positional) arguments being passed. They can be used for either optional or required arguments.

Syntax

Consider this mixin:

@mixin border-radius($value, $moz: true, $webkit: true, $standard: true) {
  @if $moz      {    -moz-border-radius: $value }
  @if $webkit   { -webkit-border-radius: $value }
  @if $standard {         border-radius: $value }
}

Sass now supports the following ways of including this mixin:

@include border-radius(10px);
@include border-radius(10px, true, false);
@include border-radius(10px, $webkit: false);
@include border-radius($value: 10px, $webkit: false);
@include border-radius($webkit: false, $value: 10px);
@include border-radius(10px, $standard: false, $webkit: false);

In addition to mixins, any functions defined by Sass or an extension to Sass can now declare that they accept named arguments. For instance, because Sass implements it’s own hsl() function so that the color may be understood and manipulated, it is possible to write:

p {
  color: hsl($hue: 180, $saturation: 78%, $lightness: 57%);
}

Since the named arguments are variable names, underscores and dashes can be used interchangeably just like elsewhere in Sass.

Documentation

The documentation of Sass functions now includes the argument names so that you may use them. Since we’ve been planning on this feature for a while now, Compass’s documentation already displays the argument names for you to use.

Improved Readability

CSS is a verbose language and for good reason: it optimizes for the reader. This is one of the ways that I think CSS is great for newcomers. Since mixins are non-standardized abstractions they are not well known to the average reader who is unfamiliar with your project. It is our hope that Sass users will take advantage of the new calling semantics to produce slightly more verbose stylesheets that are easier to understand because of the use of well-named arguments on both the calling and receiving sides of a mixin.

Argument Names are Part of Your API

It’s important to remember that your argument names will now be part of your mixin’s public API and those names should only be changed with careful consideration of the callers. Now might be a good time to change your argument names in preparation of the upcoming release. Consider using Sass’s @warn directive to deprecate argument names before you change them or to find the places where they are being used in your project.

New Power for Ruby-based Sass Functions

Sass functions defined in ruby can accept any number of positional arguments. Similarly, they can declare that they accept any keyword arguments by passing :var_kwargs => true when declaring the function to Sass. In these functions, any keyword arguments that do not get bound to named positional arguments, get passed as a Hash of name value pairs that the function can inspect and handle accordingly.
